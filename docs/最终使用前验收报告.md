# 爱木长诗最终使用前验收报告

检查日期：2026-06-18  
检查范围：本地代码库、页面注册、分包路径、云函数源码、数据库集合引用、情侣绑定链路、核心页面、特色工具、安全隐私和静态检查。  
检查方式：本轮以审计和验收为主，没有修改业务代码，没有提交 commit。

## 0. 体验化阶段更新说明

2026-06-18 追加更新：上一版报告中把 `pkg/moodDiary/moodDiary` 判断为“本地心情助手”是不再采用的方向。当前产品定位已纠正为 **小诗助手 / 恋爱 AI 小助手**：页面继续保留 AI 助手定位，预留 `aiChat` 云函数调用结构，后续由用户配置 DeepSeek / OpenAI API，API key 只放云函数环境变量，不写入前端。

本轮还完成了：

- `pages/matters/matters`：工具页改为分组、分页、搜索的小工具箱。
- `pages/remembers/remembers`：我们页下方格子改为今日计划、今日心情、今日一句话和快捷入口。
- 新增使用集合：`daily_plans`、`daily_moods`。
- 测试阶段增加本地兜底：`daily_plans` / `daily_moods` 集合未创建或权限未开放时，今日计划和心情会先保存到本地缓存，不再直接卡在失败 toast。

在暂时不处理敏感数据云函数迁移的前提下，页面体验和日常可用性可以从 C 提升到 **B-**；但最终正式使用评级仍受数据库权限和敏感集合服务端过滤影响。

## 1. 总体结论

当前版本已经从“原型”推进到“可集中测试和轻量自用”的阶段：小程序名称仍为“爱木长诗”，`app.json` 页面注册完整，`login`、`couple`、`weather` 三个云函数存在，核心情侣绑定、邀请码绑定、`coupleId` 过滤、天气 Open-Meteo 默认方案、心愿/相册/约会/纪念日/心动瞬间等页面都已有实际功能。

但它还没有达到“可以放心长期正式使用”的标准。最大风险不是编译，而是隐私和权限：树洞袜子、经期记录、心动纸条、位置等敏感数据目前仍有前端直接读写和前端过滤逻辑，正式使用前必须确认数据库权限已经收紧，最好把敏感集合迁移到云函数按 `openid + coupleId` 服务端过滤。另一个主要问题是部分工具仍是本地或占位版，例如 AI 助手不是云函数 AI，恋爱目标存在本地存储和 `love_goals` 云集合两套实现。

最终评级：**C：可以测试和轻量自用，但不建议直接作为长期正式版本使用**。如果数据库权限收紧、敏感读写迁移云函数、占位页标识清楚后，可提升到 B。

## 7. 特色工具专项检查

### 美食选择

- 已符合“今天吃什么”的随机抽取/抽卡语义。
- 支持本地新增、删除候选项、随机推荐和历史记录。
- 当前数据存在本地缓存，不按 `coupleId` 云同步。两人各自设备的候选菜单不会自动一致。
- 结论：轻量使用可达标，若想两人共同维护菜单，后续应增加云集合，例如 `food_options` 和 `food_pick_history`，并按 `coupleId` 隔离。

### 树洞袜子

- 已支持 `text/image/video/audio` 四种内容类型。
- 已支持 `public/hidden/scheduled`。
- 作者自己查询自己的全部记录；对方只查询 `public` 和 `scheduled + revealAt <= now`。
- 媒体通过 `wx.cloud.uploadFile` 存云存储 fileID，不直接把本地临时路径写进数据库。
- 风险：隐私依赖前端查询条件和数据库权限。如果数据库读权限过宽，对方或其他人可能绕过页面读取 hidden 记录。
- 结论：功能语义正确，但正式隐私强度不够，建议迁移为 `sockSecrets` 云函数，由服务端按 `ownerOpenid/coupleId/revealAt` 返回可见数据。

### 天气

- 支持“我的天气”和“Ta 的天气”。
- 不会偷偷获取对方定位；每个人需要主动同步自己的位置。
- 前端通过 `weather` 云函数请求天气，没有直接请求 Open-Meteo 或 QWeather。
- 默认 Open-Meteo，不需要 API key。
- QWeather 保留为 `WEATHER_PROVIDER=qweather` 的可选方案，key 只读云函数环境变量。
- 风险：`partner_locations` 由前端直接写入，正式权限需要限制为“只能写自己的 openid，读取最好通过云函数”。

### 经期记录

- 已是日历形式。
- 支持开始/结束、周期长度、持续天数、备注、`private/careOnly/shared`。
- 预测逻辑是基于最近一条记录的 `periodStart + cycleLength`，持续期是 `periodLength` 天。
- 页面有“关心模式”思路：对方对 `careOnly` 只拿有限字段。
- 风险：数据库权限宽时，`private` 不能真正防止被直接读取。预测说明也需要进一步强调“仅供提醒，不是医学判断”。

### 小诗助手/AI 助手

- 当前 `pkg/moodDiary/moodDiary` 已保留为“小诗助手 / 恋爱 AI 小助手”。
- 页面已有对话界面、快捷 prompt 和 `aiChat` 云函数调用结构。
- 仓库内仍没有 `cloudfunctions/aiChat`，所以真实 AI API 仍待配置。
- 未发现 OpenAI、DeepSeek 或其他 AI key 写在前端。
- 没有 `ai_sessions`、`ai_messages`、`ai_usage_logs` 的真实读写。
- 结论：AI 助手定位保留，但还不算真实接入 AI。下一步应新增 `aiChat` 云函数、环境变量、成本控制、敏感信息默认不发送的策略。
