## 1. 总体结论

这个小程序已经有一部分核心功能雏形：首页、工具页、我们页、心愿清单、约会记录、甜蜜相册、纪念日、心动瞬间、仪表盘都已经写了实际代码，并且接入了微信云数据库或云存储。但目前还不适合作为长期稳定使用版本。最大问题是情侣关系绑定没有形成可靠闭环，`coupleId` 仍以固定写死值为主，`openid` 与双方关系没有统一数据源，关键写操作大量放在前端直接访问数据库，数据库权限和数据隔离依赖云端配置但仓库内没有规则说明。建议修改顺序是：先修复登录、绑定、身份识别和数据隔离，再修复核心页面确定性 bug 和云函数，再整理工具页入口与空页面，最后做 UI 体验和长期功能。

## 3. 情侣绑定逻辑专项检查

### 当前实现

- `utils/couple.js` 中 `FIXED_COUPLE_ID = 'c_1c3d8gct77xu'`，`ensureCoupleId()` 每次都把固定值写入本地缓存。
- `utils/const.js` 也导出了同一个固定 `FIXED_COUPLE_ID`，但项目当前主要通过 `utils/couple.js` 使用。
- `pages/remembers/remembers.js` 使用云函数 `login` 获取当前 `openid`，再用 `remember_couples` 文档记录 `guoOpenid`、`luanOpenid`。
- `pages/remembers/remembers.js:115-146` 的绑定逻辑是：默认先绑定 `guoOpenid`，分享链接带 `role=luan` 时绑定 `luanOpenid`。
- `pkg/heartbeat/heartbeat.js` 又通过 `love_users` 查找同一 `coupleId` 下另一个 `openid` 作为 `partnerOpenid`。

### 当前主要问题

1. 不能稳定区分“我”和“我女朋友”。
   - 默认进入者绑定为 `guo`，分享进入者绑定为 `luan`。如果打开顺序错了、分享路径错了、缓存被清理，身份就可能错。
   - 页面写死了“小郭/小栾”的角色概念，但没有用户可确认的身份选择或确认绑定流程。

2. `openid` 使用不统一。
   - `pages/remembers/remembers.js` 用 `remember_couples.guoOpenid/luanOpenid`。
   - `pkg/heartbeat/heartbeat.js` 用 `love_users` 集合推断另一半。
   - 这会导致双方资料、纸条目标、未读统计和绑定状态不一致。

3. 缺少邀请和确认机制。
   - 没有 `invitations` 集合。
   - 没有邀请码过期、扫码确认、接受/拒绝绑定。
   - 没有“我已绑定谁”的确认界面。
   - 没有“解除绑定/重新绑定”流程。

4. 绑定不是原子操作。
   - 绑定直接在前端 `db.collection(COL).doc(coupleId).update()`。
   - 并发情况下可能两个用户同时写入同一角色，或读取旧数据后覆盖新状态。
   - 权限校验依赖数据库规则，代码本身没有服务端保证。

5. 一个用户重复绑定多个对象的问题没有处理。
   - 当前没有从 `openid` 反查用户已经属于哪个 couple。
   - 如果未来不再固定 `coupleId`，同一个用户可能加入多个 `coupleId`。
   - 即使现在固定 `coupleId`，也没有“当前账号已绑定另一角色”的清晰提示。

6. 两个人绑定状态可能不一致。
   - `remember_couples` 和 `love_users` 都能表达关系，但没有同步机制。
   - `pkg/heartbeat/heartbeat.js:127-142` 的 `upsertUser/findPartner` 可能把当前用户写到 `love_users`，但没有和 `remember_couples` 做一致性检查。

### 推荐数据结构

建议集合一：`users`

```js
{
  _id: openid,
  openid,
  nickName,
  avatarFileID,
  currentCoupleId,
  createdAt,
  updatedAt,
  lastLoginAt
}
```

建议集合二：`couples`

```js
{
  _id: coupleId,
  members: [
    { openid, role: 'me' | 'partner', displayName, avatarFileID, joinedAt }
  ],
  status: 'active' | 'unbound',
  startDate,
  createdBy,
  createdAt,
  updatedAt
}
```

建议集合三：`invitations`

```js
{
  _id: inviteId,
  coupleId,
  inviterOpenid,
  inviteeOpenid: '',
  tokenHash,
  status: 'pending' | 'accepted' | 'expired' | 'revoked',
  expiresAt,
  createdAt,
  acceptedAt
}
```

业务数据集合建议统一字段：

```js
{
  coupleId,
  createdBy,
  updatedBy,
  createdAt,
  updatedAt
}
```

### 推荐绑定流程

1. 用户首次进入小程序，调用 `login` 或 `ensureUser` 云函数，初始化 `users/{openid}`。
2. 如果用户没有 `currentCoupleId`，进入“创建情侣空间/加入邀请”状态。
3. 创建者调用 `createCoupleInvite` 云函数：
   - 创建 `couples` 文档；
   - 创建 `invitations` 文档；
   - 写入创建者为第一个 member；
   - 返回分享路径 `/pages/remembers/remembers?inviteId=xxx&token=xxx`。
4. 另一方打开分享链接，调用 `acceptInvite` 云函数：
   - 校验邀请存在、未过期、未被使用；
   - 校验当前 `openid` 未绑定其他 active couple，或要求确认切换；
   - 使用事务/原子更新把当前用户加入 `couples.members`；
   - 更新双方 `users.currentCoupleId`。
5. 页面所有业务数据通过 `currentCoupleId` 读取，不再使用写死 ID。
6. 解除绑定通过 `unbindCouple` 云函数处理，保留历史数据还是清空数据需要人工确认。

### 哪些操作必须云函数化

- 创建情侣空间。
- 生成邀请。
- 接受邀请。
- 解除绑定。
- 修改情侣开始日期和双方资料。
- 发送私密纸条。
- 删除带云存储文件的业务记录。
- 心动金库积分增减。

### 权限和数据泄露风险

- 多数查询只按 `{ coupleId }` 过滤，例如 `love_album`、`love_dates`、`love_wishes`、`love_memorials`。
- 如果数据库权限允许客户端读写，任何拿到 `coupleId` 的人都可能访问数据。
- 目前 `coupleId` 是固定字符串，公开仓库后更容易暴露。
- 写操作没有服务端校验当前 `openid` 是否属于该 `coupleId`。
- 纸条、私密纸条和相册属于隐私数据，应优先通过云函数控制。

### 可能导致读写失败的点

- 云数据库集合不存在时，多个页面没有初始化集合或明确提示。
- 需要复合索引的查询较多，例如：
  - `love_wishes`: `coupleId + status + priority + createdAt`
  - `heartbeat_notes`: `coupleId + toOpenid + read + createdAt`
  - `love_album`: `coupleId + createdAt`
  - `love_dates`: `coupleId + dateTs`
  - `love_memorials`: `coupleId + updatedAt`
- `pkg/timeline/timeline.js:251-279` 存在变量作用域 bug，会导致待办心愿统计失败。
- `pkg/heartbeat/heartbeat.js:91-92` 的 `onShow()` 直接 `refreshAll()`，如果 `onLoad()` 初始化未完成，可能用空 `coupleId/openid` 请求。
- `pages/remembers/remembers.js:27-39` 没有包裹 `login/ensureDoc/bindRole/refresh` 的整体异常处理，云函数未部署或权限不足时页面没有统一降级。

### 数据库命名问题

- `remember_couples`、`love_users`、`love_wishes`、`heartbeat_notes` 混用不同前缀。
- `remember_couples` 表示情侣关系，`love_users` 也参与关系识别，职责重叠。
- `pkg/timeline/timeline.js` 中 `COL_MEM` 指向 `love_memorials`，但项目还有一个空壳 `pkg/memorialDay`，容易混淆。
- 首页变量 `dateCount` 实际统计的是已完成心愿数，不是约会日期数。

## 6. 具体 Bug 清单

### Bug 1：分享绑定路径不存在

- 问题描述：`onShareAppMessage()` 返回 `/pages/couple/index?coupleId=${coupleId}&role=luan`，但 `app.json` 没有声明 `pages/couple/index`。
- 相关文件路径：`pages/remembers/remembers.js:275-280`，`app.json:3-7`。
- 可能原因：页面从旧路径 `pages/couple/index` 改名为 `pages/remembers/remembers` 后，分享路径未同步。
- 如何复现：在“我们”页点击邀请，右上角分享给另一个微信账号，打开分享卡片。
- 修改建议：短期改成 `/pages/remembers/remembers?coupleId=...&role=luan`；长期改为 `/pages/remembers/remembers?inviteId=...&token=...`，通过云函数接受邀请。

### Bug 2：“我们”页存在未定义事件方法

- 问题描述：WXML 中绑定了 `onMockBindPartnerAvatar`、`onMoreMoments`、`handleRecordMood`、`onQuick`、`onFab`，JS 中没有对应方法。
- 相关文件路径：`pages/remembers/remembers.wxml:41`, `:53`, `:69`, `:77-89`, `:98`，`pages/remembers/remembers.js`。
- 可能原因：页面 UI 从 mock 版本迁移到云开发版本时没有同步 JS 方法。
- 如何复现：进入“我们”页，点击“绑定情侣头像”“查看更多”“记录心情”或快捷入口。
- 修改建议：补齐方法并跳转到实际页面；未完成入口先隐藏；删除 mock 按钮或改成正式邀请绑定。

### Bug 3：仪表盘未完成心愿统计会异常

- 问题描述：`loadTodos()` 内部重复声明 `pendingWishes/pendingWishCount`，并在外层引用 `cc.total`，但 `cc` 只在内层 try 块中定义。
- 相关文件路径：`pkg/timeline/timeline.js:251-279`。
- 可能原因：复制代码时嵌套了两层 try，并没有清理变量作用域。
- 如何复现：进入仪表盘 `pkg/timeline/timeline`，观察“今日待办”的未完成心愿统计；控制台可能出现 `cc is not defined` 后被 catch。
- 修改建议：删除内层重复声明，定义 `const cc` 后直接赋值给外层 `pendingWishCount`，不要在作用域外访问。

### Bug 4：头像 fallback 文件不存在

- 问题描述：页面引用 `/images/avatar_placeholder.png`，仓库 `images/` 中没有该文件。
- 相关文件路径：`pages/remembers/remembers.wxml:15`, `pages/remembers/remembers.wxml:30`, `utils/api.js:32`。
- 可能原因：曾使用占位头像资源，但未提交或已改名。
- 如何复现：头像临时链接为空或云图片加载失败时，页面尝试显示占位头像。
- 修改建议：补一个真实占位图，或改用已有 `images/user_a.png`、`images/user_b.png`。

### Bug 5：心动瞬间初始化和 `onShow` 存在竞态

- 问题描述：`pkg/heartbeat/heartbeat.js:91-92` 的 `onShow()` 总是调用 `refreshAll()`，但 `onLoad()` 中 `ensureOpenid/upsertUser/findPartner/ensureBank` 是异步流程。
- 相关文件路径：`pkg/heartbeat/heartbeat.js:71-92`, `pkg/heartbeat/heartbeat.js:171-179`。
- 可能原因：生命周期没有用 `ready/initializing` 标志防止重复刷新。
- 如何复现：首次进入心动瞬间页，网络慢或云函数慢时，`refreshAll()` 可能先以空 openid 或未完成 bankId 执行。
- 修改建议：增加 `initialized` 状态；`onShow` 只在初始化完成后刷新；`refreshAll` 内部检查 `coupleId/openid`。

### Bug 6：私密纸条只是 UI 锁定，不是数据层锁定

- 问题描述：`loadNotes()` 会把符合条件的纸条文件 ID 转成 temp URL，再通过 `lockedForMe` 控制 UI 是否展示。前端仍可能拿到数据结构中的 URL。
- 相关文件路径：`pkg/heartbeat/heartbeat.js:531-564`。
- 可能原因：私密逻辑在前端做展示控制，没有服务端按权限返回字段。
- 如何复现：发送私密图片/语音纸条，收件人未解锁时调试前端数据。

### Bug 7：删除记录不删除云存储文件

- 问题描述：心愿、约会记录、相册、心动瞬间删除时只调用 `.remove()` 删除数据库记录。
- 相关文件路径：`pkg/wishlist/wishlist.js:330-342`, `pkg/daterecord/daterecord.js:444-453`, `pkg/photoalbum/photoalbum.js:343-352`, `pkg/heartbeat/heartbeat.js:359-368`。

- 如何复现：上传图片或语音后删除记录，再到云开发存储中查看文件。
